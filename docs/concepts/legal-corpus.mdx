# Legal Corpus

ZebraTruth's compliance pipeline doesn't fetch legal research from the web on every
request. It reads from a tenant-scoped **legal corpus** — a pre-researched store of
canonical laws plus model-generated enrichment, refreshed out-of-band by a background
worker. This page explains how the corpus works, when it refreshes, and how to force
a refresh when you need one.

## The shape of a corpus row

Each row in the corpus separates three concerns:

| Layer | Contents | Source |
|-------|----------|--------|
| **Canonical** | `lawId`, `name`, `citation`, `description`, `area`, `platform?` | Hand-curated by ZebraTruth |
| **Enrichment** | `summary`, `sourceUrl?`, `effectiveDate?`, `penaltyDescription?` | Generated by Stage A research |
| **Provenance** | `model`, `promptVersion`, `lawDefinitionHash`, `fetchedAt`, `softRefreshAt`, `expiresAt`, `source` | Set at refresh time |

The canonical block is the source of truth — when ZebraTruth edits a law definition
(e.g. a regulation amendment, a citation correction), the canonical hash changes
and the next read for that row triggers a fresh refresh automatically. No manual
cache-bust required.

## Freshness model — stale-while-revalidate

Every Stage A read applies a strict freshness policy. The goal is to serve from
the corpus as much as possible, refresh in the background when stale, and only
block on a synchronous fetch when truly necessary.

| Row state | Behavior | Latency cost |
|-----------|----------|--------------|
| **Fresh** (< 3 days) | Return immediately | Single Azure Table read |
| **Stale-but-acceptable** (3–15 days) | Return immediately + enqueue background refresh | Single Azure Table read; refresh runs out-of-band |
| **Hard expired** (> 15 days) | Synchronous refresh | Full Stage A research latency |
| **Missing** (cold tenant) | Synchronous refresh | Full Stage A research latency |
| **Hash mismatch** (canonical edited) | Synchronous refresh | Full Stage A research latency |
| **Prompt or model version mismatch** | Synchronous refresh | Full Stage A research latency |
| **Tenant-source** (custom rule) | Return immediately; never refresh | Single Azure Table read |

The 3-day soft window is a deliberate trade-off: tight enough that enrichment
reflects recent regulator guidance, loose enough that most reads serve from
cached rows.

## Refresh storm prevention

When ten concurrent reads hit the same stale row, you don't want ten background
refreshes firing — that wastes budget and risks rate-limiting. ZebraTruth uses
a row-level **lease** to make the refresh enqueue idempotent:

1. Reader sees a soft-stale row, attempts to acquire its 5-minute lease via an
   ETag-protected conditional write.
2. First reader wins, enqueues the refresh, returns the stale row.
3. Subsequent readers within the 5-minute window see the lease held, skip the
   enqueue, return the stale row anyway.
4. Worker runs the refresh; on success the entire provenance block is
   rewritten and the lease is naturally cleared.
5. If the worker dies mid-refresh, the lease expires after 5 minutes and the
   next reader can re-acquire automatically.

A second-line worker-side dedupe ensures that even if duplicate messages do
end up on the queue, only the first one performs the actual refresh — the rest
ack and skip after re-checking row freshness.

## Manual refresh

Sometimes you need fresh enrichment ahead of the soft window — a regulator
issued new guidance, you noticed an outdated detail in a check, or you simply
want to be sure the corpus reflects the latest before a critical run. Two paths:

### From the developer portal

Open the [dashboard settings page](https://developers.zebratruth.ai/dashboard/settings).
You'll see a **Legal Corpus** card with a **Refresh corpus** button. Click it
to enqueue a refresh of every stale row in your tenant. The button shows a
result panel with `requestedCount` / `enqueuedCount` / `skippedCount` / `pending`
and a "Last refreshed" timestamp that persists across reloads.

### From the API

```bash
POST https://api.zebratruth.ai/v1/corpus/refresh
Authorization: Bearer {api_key}
Content-Type: application/json

{ "scope": "all" }
```

Three scope variants are supported:

```json
// Refresh every stale row in the tenant
{ "scope": "all" }

// Refresh stale rows for one area, scoped to specific jurisdictions
{
  "scope": "area",
  "area": "ai_content",
  "jurisdictions": ["us", "eu"]
}

// Refresh exactly one row
{
  "scope": "law",
  "area": "ai_content",
  "jurisdiction": "us",
  "lawId": "ftc-ai-impersonation-rule"
}
```

**Response shape:**

```json
{
  "requestedCount": 12,   // rows the scope resolved to
  "enqueuedCount": 9,     // rows that won a lease and got queued
  "skippedCount": 3,      // lease held by recent caller, or row missing
  "pending": 0,           // over the 50-rows-per-request rate limit
  "rateLimit": 50
}
```

**Rate limits:** 50 rows per request. If `pending > 0`, the response indicates
how many rows are still pending — call again in ~1 minute to enqueue the rest.

**Lease behavior:** if you call `/corpus/refresh` twice in quick succession on
the same row, the second call returns `skippedCount: 1` rather than enqueueing
a duplicate. This makes repeated calls safe.

## Areas of the corpus

The corpus is partitioned by area-of-law. Each agent reads from its own area:

| Area | Read by | Examples |
|------|---------|----------|
| `ai_content` | AI Laws agent | EU AI Act Art. 50, California SB 942, Deep Synthesis Provisions, ELVIS Act |
| `advertising` | Advertising Law agent | FTC Endorsement Guides, UCPD, ASA codes, ASCI Code |
| `metadata` | Metadata & Labeling agent | C2PA, watermarking, provenance standards |
| `rights` | Rights Clearance agent | Right of publicity, copyright, trademark, music licensing |
| `data_privacy` | _(reserved)_ | GDPR, PIPL, CCPA/CPRA, DPDPA |
| `platform_policy` | Platform Policy agent | TikTok AIGC, YouTube synthetic content, Meta AI labels |

The `data_privacy` area is populated but no live agent currently reads it — the
slot is reserved for a future privacy-focused agent. Today, GDPR / CCPA references
that surface in compliance checks come from the AI Laws or Advertising Law agents
when they cite privacy in the context of an AI-content rule.

## Versioning and invalidation

Each refresh stamps the row's provenance with:

- **`model`** — the research model identifier (e.g. `perplexity/sonar-pro`)
- **`promptVersion`** — a versioned label for the Stage A prompt
- **`lawDefinitionHash`** — SHA-256 over the canonical block

If any of these mismatch on read, the row is treated as missing and a synchronous
refresh runs. This means:

- **Model upgrades** propagate to all rows on next read, no migration required.
- **Prompt improvements** propagate the same way.
- **Canonical edits** (law description revised, citation corrected) automatically
  invalidate the affected row's enrichment.

## What the corpus does NOT do

- **Tenant-specific framing.** Tenant rule activations, custom thresholds, and
  organization-specific compliance preferences flow into Stage B's prompt context,
  not the corpus. The corpus stays content-agnostic and focused on pure legal
  research.
- **Per-request rule synthesis.** The corpus stores already-researched citations.
  Combining multiple laws into a coherent risk assessment is Stage B's job.
- **Custom internal policies.** A future tenant-overlay feature will let
  organizations write their own internal compliance rules directly into their
  corpus partition (with `source: "tenant"` so they skip Perplexity refresh).
  This is reserved in the schema today; implementation is queued.
